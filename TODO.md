### 官方答案的特点：
+ 更像真实客服，直接给结论，不解释自己在按规则判断。
+ 对多问会逐项覆盖，但不会掉进“模板化兜底话术”。
+ 对说明书题更偏“抽取原文核心内容 + 精准图片”，不是泛化总结。
+ 对客服题更敢给确定表达，比如“24小时内揽收”“支持免费重新维修并延长质保期”这类明确承诺。
+ 句子短、信息密度高，几乎没有“取决于平台规则”“建议联系客服核实”这种保守尾巴。

### 我们当前答案
+ 太保守。大量使用“通常”“取决于平台规则”“需结合订单状态”。
+ 太像规则系统，不像客服。例如 id=1/4/6/10/16/17/21 这种，答案一开头就带规则判断口吻。
+ 多问覆盖不完整。id=3 只答了“能开发票/发票类型”，漏了“多久能收到”。
+ 有路由错误。id=40 被说明书“表带尺寸”污染，明显不是当前问题要的“换大一码/尺寸差价”。
+ 说明书题有时太啰嗦，官方样例更偏“短抽取 + 图”。

### 链路里还在严重影响质量的地方
1.路由过度把问题打到客服策略分支，直接绕开检索。
2.客服策略一次只取一个 topic，导致多意图问题漏答。
3.客服策略本身不是 RAG，而是手写规则骨架，信息上限太低。
4.证据筛选过于单产品锁定，容易错过有用 chunk。
5.说明书回答过度“格式化”，与官方抽取型答案不一致。
6.提交清洗仍然是高风险模块，会二次损伤答案。
7.当前图片链路还是“召回到什么图就附什么图”，不是“答案真正引用了哪张图就返回哪张图”。
8.src/industry_agent/agent/customer_service_kb.py 现在还是“由规则派生的内存 KB”，还不算真正独立的客服知识库；最好后续沉成单独数据源/索引。
9.src/industry_agent/rag/retriever.py 的证据筛选仍有 top_product 锁定，跨产品/混合题还可能误杀。
10.scripts/generate_submission.py 的清洗还偏“重写”，不是纯清洗，仍可能二次伤答案。
11.src/industry_agent/agent/service.py 的图片链路还是“检索到什么图就带什么图”，还没做到回答句级 grounding。
12.src/industry_agent/agent/response_formatter.py 的说明书回答仍偏模板化，离附录里的“短抽取 + 精准图片”还有差距。
13.还缺一组更完整的回归项，专门覆盖“客服多意图 + 图像问答 + 混合路由 + 追问继承”。
